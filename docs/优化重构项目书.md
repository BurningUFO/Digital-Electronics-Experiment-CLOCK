# CLOCK 项目优化重构项目书

本文档用于记录 CLOCK 项目后续优化重构的目标、讨论结论、功能方案和实现计划。

重要规则：

1. 本文档只记录已经确认要进入项目书的内容。
2. 新功能、新交互、新显示效果、新实现路线，必须由项目负责人明确同意后才能写入。
3. 讨论阶段的临时想法先保留在对话中，不默认写入本文档。
4. 若方案和当前代码行为冲突，以当前 `clock_amd` 工作区主线代码为准，先定位具体模块和信号，再决定是否修改。
5. 只围绕当前 Vivado + Nexys A7 100T 主线工程推进，不回到旧 `CLOCK` 副本。

## 1. 项目背景

当前工程已经完成基础电子钟、校时、闹钟、倒计时、OLED 模式显示、按键统一交互和蜂鸣器提醒。

后续优化重构的核心目标不是简单堆功能，而是让项目更符合 Nexys A7 100T 的板级能力，形成更成熟、更完整、更有展示价值的数字时钟系统。

## 2. 优化重构目标

已确认的首个优化方向：

1. 将 OLED 主界面从“模式名展示”升级为“智能手表式状态副屏”。
2. 不在 OLED 上重复显示当前主时间，主时间继续由数码管承担。
3. OLED 重点显示数码管不方便表达的系统状态、日期信息、提醒信息和扩展传感器信息。
4. 通过主 UI 改造提升项目展示效果和完整度，让系统更像一个可实际使用的数字时钟设备。

## 3. 设计原则

待确认。

## 4. 已确认功能方案

### 4.1 OLED 智能手表式主 UI

已确认将 OLED 主界面设计为智能手表式状态副屏。

设计取舍：

1. 数码管已经承担当前时间主显示，因此 OLED 不再重复放大显示 `HH:MM:SS`。
2. OLED 用于展示更适合点阵屏表达的辅助信息，包括日期、星期、温度、最近计划点、倒计时状态和最近闹钟。
3. 主 UI 保持信息分区清晰，不把所有内容平均铺满；优先显示最需要用户注意的下一事件和正在运行的倒计时。
4. 温度显示作为后续亮点功能，优先考虑接入 Nexys A7 板载 ADT7420 温度传感器。
5. 麦克风等板载资源暂不进入首版主 UI，可作为后续特色交互方向继续讨论。

建议主界面结构：

```text
05/10 SUN  26C

NEXT CLASS
15:40  +01:22

AL 07:30  CT 00:12
```

区域含义：

1. 顶部状态栏：日期、星期、温度。
2. 中部主信息区：最近计划点，例如下一节课、下一次作息提醒。
3. 底部状态区：最近闹钟与倒计时状态。

信息优先级：

1. 正在运行的倒计时。
2. 最近计划点。
3. 最近启用闹钟。
4. 日期和星期。
5. 温度。

当倒计时运行时，底部优先显示倒计时剩余时间；没有倒计时时，底部可显示最近闹钟或计划状态。

### 4.2 八位数码管显示与 SW0 设置层

已确认对八位数码管进行统一职责划分，增强板级状态展示能力。

数码管从左到右定义为：

```text
D7 D6  D5 D4  D3 D2  D1 D0
模式 状态  时十 时个  分十 分个  秒十 秒个
```

对应 Nexys A7 八位数码管：

```text
AN7 AN6  AN5 AN4  AN3 AN2  AN1 AN0
```

显示原则：

1. D7 显示当前模式。
2. D6 显示当前模式下的状态。
3. D5~D0 优先保持三组两位数字结构，用于显示时间、闹钟时间、倒计时或日期数据。
4. 数码管承担快速状态识别和核心数字显示，OLED 承担更丰富的状态说明。

模式位 D7 规划：

| 模式 | D7 显示 | 说明 |
| --- | --- | --- |
| CLOCK | `n` | normal / clock，避免和 COUNT 的 `C` 冲突 |
| TIME | `t` | 当前时间设置 |
| ALARM | `A` | 闹钟 |
| HOUR | `H` | 12/24 小时制 |
| COUNT | `C` | 倒计时 |
| SCHED | `S` | 作息 / 计划 |

状态位 D6 规划：

| D6 显示 | 含义 |
| --- | --- |
| 空白 | 普通浏览 |
| `o` | 当前功能或当前槽位开启 |
| `F` | 当前功能或当前槽位关闭 |
| `r` | 倒计时运行中 |
| `P` | 暂停 / 停止 |
| `d` | 完成 / 日期页 |
| `1` | 12 小时制 |
| `2` | 24 小时制 |

已确认 `SW0` 作为所有模式的统一设置层开关：

```text
SW0 = 0：主浏览层
SW0 = 1：当前模式的详情 / 查看 / 设置层
```

基础交互原则：

1. `SW0 = 0` 时，系统处于浏览层，优先用于快速查看当前模式状态。
2. `SW0 = 1` 时，系统进入当前模式的详情或设置层。
3. `BTNL / BTNR` 在设置层中用于切换字段、槽位或条目。
4. `BTNU / BTND` 在设置层中用于修改当前字段。
5. `BTNC` 在设置层中用于确认、保存、启停或开关，具体含义由当前模式定义。
6. `SW0` 直接决定是否处于设置层，不需要再按 `BTNC` 二次进入编辑。
7. `BTNC` 不再承担进入 / 退出编辑态的职责，后续释放给确认、开关、启停、保存或消音等上下文动作。

全局交互规则：

| 层级 | BTNL / BTNR | BTNU / BTND | BTNC |
| --- | --- | --- | --- |
| `SW0 = 0` 浏览层 | 切换模式 | 当前模式快捷操作 | 确认 / 消音 / 快捷动作 |
| `SW0 = 1` 设置层 | 切换字段、槽位或条目 | 修改当前字段 | 确认 / 开关 / 启停 / 保存 |

各模式字段顺序：

| 模式 | `SW0 = 1` 设置层字段顺序 |
| --- | --- |
| CLOCK | 月份 -> 日期 -> 星期 |
| TIME | 小时 -> 分钟 -> 秒 |
| ALARM | 闹钟编号 -> 小时 -> 分钟 -> 秒 -> 开关 |
| HOUR | 12/24 小时制 |
| COUNT | 小时 -> 分钟 -> 秒 |
| SCHED | 计划编号 -> 开关 |

### 4.3 CLOCK 详情页

已确认 `CLOCK + SW0 = 1` 不重复承担当前时间设置，当前时间设置仍由 `TIME` 模式负责。

`CLOCK + SW0 = 1` 定义为日期 / 星期设置页，用于给 OLED 智能手表式主 UI 提供日期和星期数据。

浏览层：

```text
n _  HH MM SS
```

设置层：

```text
n d  MM DD WW
```

字段含义：

1. `MM`：月份。
2. `DD`：日期。
3. `WW`：星期编号。

建议星期编号：

| 显示 | 含义 |
| --- | --- |
| `01` | Monday |
| `02` | Tuesday |
| `03` | Wednesday |
| `04` | Thursday |
| `05` | Friday |
| `06` | Saturday |
| `07` | Sunday |

设置层交互：

1. `BTNL / BTNR`：切换月份、日期、星期字段。
2. `BTNU / BTND`：当前字段加减。
3. `BTNC`：确认 / 保存。
4. 当前选中字段使用闪烁提示。

### 4.4 ALARM 八槽位显示与 LED 指示

已确认 ALARM 模式支持 8 个闹钟槽位作为优化目标。

ALARM 首版数据模型：

1. 系统保存 8 个闹钟槽位。
2. 每个槽位保存 `HH:MM:SS` 和开启状态。
3. 复位后 8 个槽位默认全部关闭。
4. 复位后 8 个槽位时间默认均为 `00:00:00`。
5. 每个槽位最多保留一个当前有效的贪睡临时提醒。
6. 贪睡临时提醒不修改原槽位时间。

在 ALARM 模式下，只要 D7 显示 `A`，LED0~LED7 就用于表示 8 个闹钟槽位状态：

| LED 状态 | 含义 |
| --- | --- |
| 常亮 | 对应闹钟槽位开启 |
| 熄灭 | 对应闹钟槽位关闭 |
| 闪烁 | 当前正在查看 / 编辑的闹钟槽位 |

若某个槽位同时处于开启和当前选中状态，闪烁优先。

ALARM 浏览层：

```text
A o  HH MM SS
```

或：

```text
A F  HH MM SS
```

含义：

1. D6 显示最近启用闹钟或当前槽位的开启状态。
2. `o` 表示开启，`F` 表示关闭。
3. D5~D0 显示闹钟时间。
4. `SW0 = 0` 浏览层优先显示最近一个已开启闹钟。
5. 若没有任何闹钟开启，则显示 `A F 00 00 00`。

ALARM 设置层：

```text
A o  HH MM SS
```

或：

```text
A F  HH MM SS
```

已确认 D6 在 ALARM 中显示当前槽位的 `o/F` 状态，不使用 `E` 表示编辑态。

ALARM 设置层字段建议：

```text
闹钟编号 -> 小时 -> 分钟 -> 秒 -> 使能
```

字段提示：

1. 选中闹钟编号：对应 LED 闪烁。
2. 选中小时：小时两位闪烁。
3. 选中分钟：分钟两位闪烁。
4. 选中秒：秒两位闪烁。
5. 选中使能：D6 的 `o/F` 闪烁。

同秒多闹钟触发规则：

1. 若多个闹钟槽位在同一秒触发，按槽位编号从小到大排队处理。
2. 最小编号槽位优先进入统一提醒系统并显示闹钟弹窗。
3. 用户通过 `BTNC` 关闭本次提醒或方向键选择贪睡后，系统继续显示下一个待处理的更高编号槽位。
4. 该规则同样适用于同一秒内多个贪睡临时提醒触发的情况。
5. 若同一时间既有普通闹钟触发又有贪睡临时提醒触发，仍按对应槽位编号从小到大处理。

### 4.5 SCHED 固定八计划点

已确认 SCHED 首版采用固定 8 个计划点，不在首版支持编辑计划时间。

首版功能范围：

1. 系统内置 8 个固定计划点。
2. 每个计划点支持开启 / 关闭。
3. LED0~LED7 在 SCHED 模式下用于表示 8 个计划点状态。
4. OLED 智能手表式主 UI 显示最近一个已开启的计划点。
5. 后续可在统一提醒系统中接入计划点到点提醒。

固定计划点：

| 编号 | 时间 | OLED 名称 | 类型 |
| --- | --- | --- | --- |
| 0 | `08:00:00` | `CLASS 1` | 上课 |
| 1 | `09:40:00` | `BREAK` | 课间 |
| 2 | `10:00:00` | `CLASS 2` | 上课 |
| 3 | `11:40:00` | `LUNCH` | 午休 |
| 4 | `14:00:00` | `CLASS 3` | 上课 |
| 5 | `15:40:00` | `BREAK` | 课间 |
| 6 | `19:00:00` | `STUDY` | 晚自习 |
| 7 | `21:30:00` | `REST` | 休息 |

SCHED 模式下 LED 规则：

| LED 状态 | 含义 |
| --- | --- |
| 常亮 | 对应计划点开启 |
| 熄灭 | 对应计划点关闭 |
| 闪烁 | 当前正在查看 / 设置的计划点 |

若某个计划点同时处于开启和当前选中状态，闪烁优先。

SCHED 浏览层：

```text
S o  HH MM SS
```

或：

```text
S F  00 00 00
```

含义：

1. 有已开启的后续计划点时，D6 显示 `o`，D5~D0 显示最近计划点时间。
2. 没有可用计划点时，D6 显示 `F`，D5~D0 可显示 `00:00:00`。
3. 浏览层显示 24 小时循环内最近一个已开启计划点。

SCHED 设置层：

```text
S o  HH MM SS
```

或：

```text
S F  HH MM SS
```

含义：

1. D5~D0 显示当前查看的固定计划点时间。
2. D6 显示当前计划点开启状态，`o` 表示开启，`F` 表示关闭。

设置层字段顺序：

```text
计划编号 -> 开关
```

设置层交互：

1. `BTNL / BTNR`：切换计划编号或开关字段。
2. `BTNU / BTND`：切换计划点或修改开关状态。
3. `BTNC`：切换当前计划点开启 / 关闭。
4. 选中计划编号时，对应 LED 闪烁。
5. 选中开关时，D6 的 `o/F` 闪烁。

OLED 首页显示规则：

```text
NEXT <NAME>
HH:MM  +HH:MM
```

示例：

```text
NEXT CLASS 3
15:40  +01:22
```

其中 `HH:MM` 表示最近计划点时间，`+HH:MM` 表示距离该计划点还有多久。若首版实现距离计算压力较大，可先显示计划名称和时间，后续再补距离显示。

同秒多计划点触发规则：

1. 正常固定计划表中不主动设计同秒重复计划点。
2. 若后续扩展导致多个计划点同一秒触发，按计划编号从小到大依次提醒。
3. 用户确认当前计划提醒后，再显示下一个更高编号计划提醒。

### 4.6 统一提醒系统与闹钟贪睡

已确认后续新增统一提醒系统，用于集中处理倒计时结束、闹钟到点和计划提醒。

提醒优先级：

```text
倒计时结束 > 闹钟 > 计划提醒
```

优先级规则：

1. 多个提醒同时发生时，优先显示和播放最高优先级提醒。
2. 高优先级提醒可以抢占低优先级提醒。
3. 低优先级提醒不丢失，可作为待处理提醒保留。
4. 用户确认当前提醒后，若仍有待处理提醒，系统继续显示下一条最高优先级提醒。

蜂鸣器节奏：

| 事件 | 蜂鸣器节奏 | 预计响铃时长 |
| --- | --- | --- |
| 倒计时结束 | `100ms` 响 / `100ms` 停，重复 8 次 | 约 `1.6s` |
| 闹钟 | `120ms` 响 / `120ms` 停，3 次一组，共 3 组，组间停 `500ms` | 约 `3s` |
| 计划提醒 | `150ms` 响 / `250ms` 停，重复 2 次 | 约 `0.8s` |

OLED 提醒显示：

1. 正常情况下，OLED 显示智能手表式主 UI、当前模式页或设置页。
2. 当 `notify_active = 1` 时，OLED 临时覆盖原页面，显示全屏提醒弹窗。
3. 提醒弹窗使用事件标题、核心时间信息和按键提示。
4. 弹窗边框闪烁，标题可闪烁或反白，用于增强提示效果。
5. 当前提醒被确认或自动结束后，OLED 恢复原页面。

弹窗示例：

```text
+------------+
| COUNT DONE |
|  00:00:00  |
|  BTNC MUTE |
+------------+
```

```text
+------------+
|   ALARM    |
|  07:30:00  |
| C CLOSE    |
| U+1 R+3    |
| D+5 L+10   |
+------------+
```

```text
+------------+
| NEXT CLASS |
|   15:40    |
|  BTNC OK   |
+------------+
```

OLED 弹窗保持规则：

| 事件 | 是否覆盖 OLED | 是否自动消失 |
| --- | --- | --- |
| 倒计时结束 | 是 | 不自动消失，需 `BTNC` 确认 |
| 闹钟 | 是 | 不自动消失，需 `BTNC` 关闭或方向键贪睡 |
| 计划提醒 | 是 | 可 `10s` 自动消失，也可 `BTNC` 提前确认 |

全局 `BTNC` 规则：

1. 只要当前有提醒弹窗或蜂鸣器正在工作，`BTNC` 优先作为提醒确认 / 消音键。
2. `BTNC` 在提醒激活时不执行当前模式下的确认、开关、启停或保存动作。
3. 按下 `BTNC` 后，当前提醒停止蜂鸣器并清除弹窗。
4. 若仍有其他待处理提醒，系统继续按优先级显示下一条提醒。
5. 只有在没有任何提醒激活时，`BTNC` 才恢复为当前模式的上下文功能。

闹钟贪睡规则：

1. 贪睡功能只在 `notify_type = ALARM` 且闹钟弹窗激活时生效。
2. 闹钟弹窗期间，方向键被临时解释为贪睡时间选择。
3. `BTNC` 表示关闭本次响铃 / 清除当前闹钟弹窗。
4. 贪睡不修改原闹钟槽位时间，只生成一次临时延后提醒。
5. 原闹钟槽位仍保持开启状态，后续日期或下一轮触发仍按原设定时间工作。
6. 贪睡提醒再次触发时仍属于 `ALARM` 类型，优先级低于倒计时结束，高于计划提醒。
7. 同一个闹钟槽位只保留一个当前有效的贪睡时间；再次贪睡时更新该临时提醒时间。
8. 贪睡时间需要支持跨小时和跨天回绕，例如 `23:58 + 5min -> 00:03`。

闹钟弹窗按键：

| 按键 | 动作 |
| --- | --- |
| `BTNC` | 关闭本次闹钟提醒，不关闭原闹钟槽位 |
| `BTNU` | 贪睡 1 分钟 |
| `BTNR` | 贪睡 3 分钟 |
| `BTND` | 贪睡 5 分钟 |
| `BTNL` | 贪睡 10 分钟 |

### 4.7 date_core 日期 / 星期模块

已确认新增 `date_core` 作为首版软日历模块，用于支持 CLOCK 日期设置页和 OLED 智能手表式主 UI。

首版功能范围：

1. 保存月份、日期和星期。
2. 支持手动设置月份、日期和星期。
3. 支持每天自动进一天。
4. 不保存年份。
5. 不处理闰年。
6. 2 月固定为 28 天。
7. 不提供断电保持，复位后回到默认日期。

默认值：

```text
01/01 MON
```

内部范围：

| 字段 | 范围 |
| --- | --- |
| 月份 | `01~12` |
| 日期 | 根据当前月份限制 |
| 星期 | `01~07` |

星期编号：

| 编号 | OLED 显示 |
| --- | --- |
| `01` | `MON` |
| `02` | `TUE` |
| `03` | `WED` |
| `04` | `THU` |
| `05` | `FRI` |
| `06` | `SAT` |
| `07` | `SUN` |

月份天数：

| 月份 | 最大日期 |
| --- | --- |
| 1, 3, 5, 7, 8, 10, 12 | 31 |
| 4, 6, 9, 11 | 30 |
| 2 | 28 |

自动进日规则：

1. 当前时间从 `23:59:59` 跳到 `00:00:00` 时，日期自动加一天。
2. 星期同步加一，`SUN -> MON`。
3. 日期超过当前月份最大日期时，日期回到 `01`，月份加一。
4. 月份超过 `12` 时，月份回到 `01`。

手动设置规则：

1. `CLOCK + SW0 = 1` 进入日期 / 星期设置页。
2. 字段顺序为：月份 -> 日期 -> 星期。
3. `BTNU / BTND` 对当前字段加减。
4. 月份、日期和星期均采用循环调整。
5. 当月份变化导致当前日期非法时，日期自动夹到当前月份最大日期。

示例：

```text
03/31 -> 修改月份到 02 -> 02/28
```

### 4.8 HOUR 12/24 小时制

已确认 `HOUR` 模式首版只管理全局时间显示格式，不改变系统内部时间保存和事件比较方式。

核心原则：

```text
内部永远使用 24 小时制；12/24 小时制只影响显示。
```

首版功能范围：

1. 新增全局显示格式设置：24 小时制 / 12 小时制。
2. `HOUR + SW0 = 0` 浏览当前显示格式。
3. `HOUR + SW0 = 1` 设置当前显示格式。
4. 切换 12/24 小时制时，不修改当前时间、闹钟时间、计划时间或倒计时时长。
5. AM/PM 首版只在 OLED 中显示，不强行塞进数码管。

数码管显示：

浏览层：

```text
H 2  HH MM SS
```

或：

```text
H 1  HH MM SS
```

设置层：

```text
H 2  HH MM SS
```

或：

```text
H 1  HH MM SS
```

含义：

1. D7 显示 `H`。
2. D6 显示 `2` 表示 24 小时制。
3. D6 显示 `1` 表示 12 小时制。
4. D5~D0 显示当前时间，并按当前显示格式转换。

设置层交互：

1. 设置层字段只有一个：12/24 小时制。
2. `BTNU / BTND / BTNC` 均可用于切换 12/24 小时制。
3. `BTNL / BTNR` 无字段可切，首版可保留无动作。

12 小时制转换规则：

| 内部 24 小时制 | 12 小时显示 | AM/PM |
| --- | --- | --- |
| `00:xx:xx` | `12:xx:xx` | AM |
| `01~11` | `01~11` | AM |
| `12:xx:xx` | `12:xx:xx` | PM |
| `13~23` | `01~11` | PM |

对各功能的影响：

1. `time_core` 内部继续保存 `00~23`。
2. `TIME` 设置页始终使用 24 小时制编辑，避免 12 小时制下缺少 AM/PM 字段导致歧义。
3. `alarm_ctrl` 内部继续用 24 小时制保存和比较。
4. `ALARM` 设置页始终使用 24 小时制编辑。
5. `schedule_ctrl` 固定计划点内部继续使用 24 小时制。
6. `COUNT` 是时长，不受 12/24 小时制影响。
7. `notification_ctrl` 的事件比较不受 12/24 小时制影响。
8. OLED 首页、OLED 弹窗和浏览层显示可按当前显示格式显示，并在 12 小时制下显示 AM/PM。

避免冲突的约束：

1. 12/24 小时制只作为显示格式，不参与事件比较。
2. 切换显示格式不会改变任何已保存时间。
3. 所有需要显示“当天时间点”的模块共用同一套小时转换逻辑。
4. 倒计时显示不经过 12/24 小时制转换。

### 4.9 温度显示

已确认温度显示作为首版正式目标，用于增强 OLED 智能手表式主 UI 的板级展示能力。

功能定位：

1. 温度显示是相对独立的辅助信息功能。
2. 温度数据优先来自 Nexys A7 板载 ADT7420 温度传感器。
3. 温度显示只进入 OLED 状态栏，不影响数码管主显示。
4. 温度读取失败不影响时钟、闹钟、倒计时、计划提醒和蜂鸣器。

OLED 显示格式：

```text
05/13 WED 26C
```

读取失败或传感器不可用时：

```text
05/13 WED --C
```

显示规则：

1. 首版只显示整数摄氏度。
2. 不显示小数，降低 OLED 占用和逻辑复杂度。
3. 温度建议每 `1s~2s` 更新一次。
4. 温度异常或 I2C 读取失败时显示 `--C`。

实现分阶段：

1. 第一阶段：OLED 主 UI 预留温度显示接口，可先显示固定演示值或 `--C`。
2. 第二阶段：新增 ADT7420 读取模块，接入真实板载温度数据。
3. 温度模块应独立于提醒系统和时间核心，避免传感器异常拖垮主功能。

板级连接确认：

1. 当前外接 OLED 使用 Pmod JB 的 `JB1 / D14` 作为 `OLED_SCL`，`JB2 / F16` 作为 `OLED_SDA`。
2. Nexys A7 板载 ADT7420 温度传感器使用独立信号 `TMP_SCL / TMP_SDA`。
3. Digilent Nexys A7 Master XDC 中，`TMP_SCL` 对应 FPGA 引脚 `C14`，`TMP_SDA` 对应 FPGA 引脚 `C15`。
4. 因此当前设计中，OLED I2C 与板载 ADT7420 I2C 不共用同一对 SCL/SDA。
5. 首版可为 ADT7420 单独实例化温度 I2C 读取模块，不需要和 OLED I2C 做总线仲裁。
6. ADT7420 的 I2C 从设备地址为 `0x4B`。

## 8. 显示与交互设计

### 8.1 OLED 主 UI 初步实现思路

首版主 UI 以静态清晰为主，优先完成信息层级和状态表达，不优先追求复杂动画。

初步实现方向：

1. 扩展 `oled_ui_display.v` 的文本绘制能力，使其支持数字、冒号、斜杠、星期缩写和状态缩写。
2. 从 `clock.v` 向 OLED 显示层传入当前日期、星期、温度、闹钟状态、倒计时状态和最近计划点状态。
3. 日期和星期可先由软日历模块提供，复位后需要手动设置或后续由外部接口初始化。
4. 温度数据后续由 ADT7420 读取模块提供；在传感器链路完成前，可先预留显示接口。
5. 当前模式切换动画可以保留，但主界面应以稳定、易读、设备化的信息布局为主。

首版不优先实现：

1. OLED 上重复显示大号主时间。

```verilog
module date_core(
```


实现原则：

1. 温度读取模块独立运行，不参与主时间链路。
2. 温度读取失败时，`temp_valid = 0`，OLED 显示 `--C`。
3. 温度读取成功时，OLED 显示整数摄氏度。
4. 读取频率不需要高，建议 `1s~2s` 一次。
5. 当前已确认 ADT7420 与外接 OLED 不在同一 I2C 总线上，可使用独立 I2C 读取模块。
6. 温度模块顶层端口建议使用 `TMP_SCL` 和 `TMP_SDA`，并在 `.xdc` 中约束到 `C14 / C15`。
7. 可按 ADT7420 上电默认状态读取温度寄存器，两字节温度值右移 3 位后按 `0.0625 C/LSB` 换算。
8. 首版 OLED 只显示整数摄氏度，换算结果可截断或四舍五入，具体取舍在实现阶段确定。

## 10. 决策记录

### [2026-05-10] OLED 主界面定位

- 决策：OLED 主界面定位为智能手表式状态副屏，不重复显示数码管已经显示的当前主时间。
- 原因：数码管更适合作为 `HH:MM:SS` 主显示，OLED 更适合展示日期、星期、温度、计划、闹钟和倒计时等状态信息。
- 影响范围：后续 UI 改造优先关注 `oled_ui_display.v`、`clock.v`、未来日期/星期模块、温度读取模块和计划提醒模块。
- 状态：已确认。

### [2026-05-10] 数码管与 SW0 设置层定位

- 决策：八位数码管采用 `模式 + 状态 + 六位数字` 的统一结构；`SW0 = 1` 作为所有模式的设置层入口。
- 原因：充分利用 Nexys A7 的 8 位数码管和开关资源，让模式、状态和核心数字信息在板上直接可见。
- 影响范围：后续需要改造 `display_ctrl.v`、`nexys_seg_scan.v`、`seg_7.v`、`clock_amd_top.v`、`clock.v` 和 `.xdc` 约束。
- 状态：已确认。

### [2026-05-10] ALARM 八槽位与 LED 指示

- 决策：ALARM 模式优化目标为 8 个闹钟槽位；当 D7 显示 `A` 时，LED0~LED7 表示 8 个闹钟槽位的开关和选中状态。
- 原因：LED 与闹钟槽位一一对应，展示直观，也能体现板级资源利用。
- 影响范围：后续需要扩展 `alarm_ctrl.v` 的存储结构、比较逻辑、字段选择逻辑和 LED 输出逻辑。
- 状态：已确认。

### [2026-05-10] 全局交互规则和字段顺序

- 决策：采用 `SW0 = 0` 浏览层、`SW0 = 1` 设置层的全局交互结构；各模式设置层字段顺序按 CLOCK、TIME、ALARM、HOUR、COUNT 先行确定，SCHED 后续单独确认。
- 原因：先统一交互状态机，再推进显示、闹钟多槽位、OLED 和提醒系统重构，避免后续模块接口反复变动。
- 影响范围：后续重点影响 `ui_ctrl.v`、`clock.v`、`display_ctrl.v`、`alarm_ctrl.v`、`countdown_ctrl.v` 和未来 SCHED 模块。
- 状态：已确认。

### [2026-05-10] SW0 直接决定设置层

- 决策：`SW0 = 1` 时直接进入当前模式的设置层，不需要再按 `BTNC` 二次进入编辑。
- 原因：`SW0` 是有物理状态的开关，适合直接表达“当前处于设置层”；释放 `BTNC` 后，可将其用于确认、开关、启停、保存和消音等更有价值的上下文操作。
- 影响范围：后续 `ui_ctrl.v` 中的 `edit_active` 逻辑应重构为由 `SW0` 直接驱动的 `setting_active`，并重新分配 `BTNC` 输出脉冲含义。
- 状态：已确认。

### [2026-05-13] SCHED 首版范围

- 决策：SCHED 首版采用固定 8 个计划点，每个计划点可开启 / 关闭；LED0~LED7 表示计划点状态，OLED 首页显示最近一个已开启计划点。
- 原因：该方案比完全可编辑计划表实现风险低，同时能补齐 OLED 主 UI 中“最近计划点”的数据来源，并充分利用板载 LED。
- 影响范围：后续需要新增 SCHED 控制模块，并接入数码管显示、LED 指示、OLED 首页和统一提醒系统。
- 状态：已确认。

### [2026-05-13] 统一提醒系统与闹钟贪睡

- 决策：统一提醒优先级为倒计时结束、闹钟、计划提醒；OLED 使用覆盖式提醒弹窗；`BTNC` 在提醒激活时全局作为确认 / 消音键；闹钟弹窗支持方向键贪睡。
- 原因：统一仲裁可以避免多事件同时触发时行为混乱；贪睡功能只对闹钟生效，不修改原闹钟时间，可以增强功能完整度且避免破坏闹钟槽位设置。
- 影响范围：后续需要新增 `notification_ctrl`，重构蜂鸣器控制路径，扩展 OLED 弹窗显示，并让 `alarm_ctrl.v` 支持闹钟槽位来源和临时贪睡提醒。
- 状态：已确认。

### [2026-05-13] date_core 首版范围

- 决策：新增 `date_core` 软日历模块，首版只保存月份、日期和星期，不保存年份，不处理闰年，2 月固定 28 天，复位默认 `01/01 MON`。
- 原因：日期 / 星期是 OLED 智能手表式主 UI 和 CLOCK 设置页的必要数据，但首版不需要完整万年历复杂度。
- 影响范围：后续需要新增 `date_core.v`，并接入 `clock.v`、`display_ctrl.v`、`oled_ui_display.v` 和 CLOCK 设置层交互。
- 状态：已确认。

### [2026-05-13] ALARM 八槽位数据模型

- 决策：8 个闹钟槽位复位默认全关闭，时间默认 `00:00:00`；浏览层显示最近一个已开启闹钟，设置层显示当前选中槽位；每个槽位最多保留一个贪睡临时提醒。
- 决策：多个闹钟在同一秒触发时，按槽位编号从小到大依次进入提醒弹窗；用户确认或贪睡当前槽位后，再显示下一个更高编号槽位。
- 原因：该规则可预测、易实现，且避免多个同秒闹钟互相覆盖或丢失。
- 影响范围：后续 `alarm_ctrl.v` 需要支持 8 槽位存储、最近启用闹钟选择、同秒触发队列、槽位编号输出和每槽位贪睡临时提醒。
- 状态：已确认。

### [2026-05-13] SCHED 固定计划点数据

- 决策：SCHED 首版固定 8 个计划点：`08:00 CLASS 1`、`09:40 BREAK`、`10:00 CLASS 2`、`11:40 LUNCH`、`14:00 CLASS 3`、`15:40 BREAK`、`19:00 STUDY`、`21:30 REST`。
- 决策：SCHED 浏览层显示 24 小时循环内最近一个已开启计划点；OLED 首页显示 `NEXT <NAME>`、计划时间和距离该计划点的剩余时间。
- 原因：固定计划点能以较低实现风险补齐计划提醒功能，并为 OLED 首页提供明确数据来源。
- 影响范围：后续 `schedule_ctrl.v` 需要内置固定计划表、计划开关状态、最近计划选择、同秒编号排队提醒和 OLED 名称输出。
- 状态：已确认。

### [2026-05-13] HOUR 12/24 小时制范围

- 决策：`HOUR` 模式只管理显示格式；系统内部时间保存、闹钟比较、计划比较和倒计时均不受 12/24 小时制影响。
- 决策：`TIME` 和 `ALARM` 设置页始终使用 24 小时制编辑；AM/PM 首版只在 OLED 显示，不写入数码管。
- 原因：该方案稳定、清楚、实现风险低，可避免 12 小时制编辑时缺少 AM/PM 字段导致歧义，也避免影响提醒比较逻辑。
- 影响范围：后续需要新增 `hour_format_ctrl.v` 和统一小时显示转换逻辑，并接入 `display_ctrl.v`、`oled_ui_display.v` 和提醒弹窗显示。
- 状态：已确认。

### [2026-05-13] 温度显示首版目标

- 决策：温度显示纳入首版正式目标，优先使用 Nexys A7 板载 ADT7420 作为真实温度数据源。
- 决策：OLED 状态栏显示整数摄氏度，读取失败时显示 `--C`；温度模块异常不影响主时钟、提醒和蜂鸣器功能。
- 决策：当前外接 OLED 使用 `D14/F16`，板载 ADT7420 使用 `TMP_SCL=C14 / TMP_SDA=C15`，两者不共享同一 I2C 总线；首版 ADT7420 可使用独立 I2C 读取模块。
- 原因：温度显示是相对独立的板级资源利用亮点，能增强 OLED 智能手表式主 UI 的完整度。
- 影响范围：后续需要新增 `adt7420_reader.v` 或等效温度读取模块，并向 `oled_ui_display.v` 提供 `temp_valid / temp_c`。
- 状态：已确认。

- 影响范围：后续代码实施、测试验收和文档同步均按该分期路线组织。
