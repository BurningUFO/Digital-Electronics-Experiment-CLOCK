## 1. 项目概述：从“电子钟实验”到“时间管理产品”

### 1.1 项目背景

这一节不要一上来讲 Verilog，而是先讲“为什么做这个产品”。

可以这样写：

本项目最初目标是完成数字电子钟的基本功能，包括时、分、秒显示，系统时钟分频，计数控制，复位和校时。但在完成基础要求后，我们进一步将它扩展为一个更加接近实际产品形态的“智能时间管理终端”。它不仅能独立显示和维护时间，还能管理闹钟、倒计时、课程/日程提醒，并通过 USB-UART 与 PC 端 ClockLink Studio 进行通信。

### 1.2 产品定位

建议写成三句话：

第一，它是一块 **独立可运行的 FPGA 电子钟**，不依赖 PC 即可完成基本计时、显示、校时和提醒。

第二，它是一套 **多模式时间管理工具**，把闹钟、倒计时、计划提醒、12/24 小时制显示整合到统一交互逻辑中。

第三，它也是一个 **软硬件联动系统**，通过 COMM 模式和 ClockLink Studio，让 PC 可以发送消息、同步时间、设置闹钟/日程/倒计时。

### 1.3 平台与载体

说明硬件平台：

本项目运行在 `Nexys A7-100T` 开发板上，开发板核心器件为 `XC7A100T-1CSG324C`，主开发工具为 Vivado，板载主时钟为 `100 MHz`，常用资源包括拨码开关、方向按键、LED、八位七段数码管以及 USB-JTAG / USB-UART 等。

## 3. 产品整体架构

### 3.1 系统组成

建议把系统分成四层来写：

**第一层：输入交互层**

包括 `BTNL / BTNR / BTNU / BTND / BTNC` 五个方向/中心按键、`SW[15:0]` 拨码开关、`CPU_RESETN` 复位键。

**第二层：核心逻辑层**

包括时间核心、日期核心、模式控制、校时控制、闹钟控制、倒计时控制、日程控制、提醒仲裁、通信控制等模块。

**第三层：输出显示层**

包括八位七段数码管、LED 状态指示、外接蜂鸣器、OLED 状态副屏。

**第四层：PC 通信层**

包括 USB-UART、ClockLink ASCII 协议、PC 端 ClockLink Studio GUI / CLI。

仓库中的模块说明也已经把当前工程划分为板级适配与显示模块、功能主线模块、通信模块等；顶层 `clock_amd_top.v` 负责板级时钟、按键、开关、LED、数码管、OLED、ADT7420、蜂鸣器和 USB-UART 映射，`clock.v` 负责主线功能集成。

## 4. 基础功能介绍：一块真正可用的数字电子钟

这一章专门讲课程基础要求，不要被扩展功能淹没。

### 4.1 基本时间显示

写作重点：

项目在 `CLOCK` 模式下显示当前时间，采用 `HH:MM:SS` 结构组织时、分、秒。数码管作为主显示区域，OLED 可作为状态副屏辅助展示日期、温度和当前模式。八位七段数码管在 Nexys A7 上是共阳极、低有效并且需要动态扫描，因此显示逻辑被拆分为“显示内容生成”和“数码管扫描驱动”两部分。

可以写成产品化语言：

> 用户打开系统后，最先看到的是一个稳定运行的数字时钟。它不是简单地把数值输出到灯上，而是通过动态扫描把时、分、秒组织成稳定、连续、可读的主界面。

### 4.2 系统时钟分频与计数控制

写作重点：

说明 FPGA 板载输入是 `100 MHz`，而人能感知的电子钟需要 `1 Hz` 走时，因此必须进行分频或时钟使能设计。项目采用统一 `CLK100MHZ` 时钟域，再通过 `tick_1k` 和 `1Hz` 使能来驱动低速逻辑，这样可以减少多时钟域带来的复杂度。

可以强调课程知识：

> 这一部分对应课程中的时序逻辑、计数器、分频器、时钟使能和同步设计。我们没有把系统拆成多个杂乱时钟，而是尽量保持单主时钟域，通过使能信号控制不同模块的更新节奏。

### 4.3 复位功能

写作重点：

`CPU_RESETN` 是低有效复位，按下时系统回到初始状态。文档中要说明复位的意义不是“清零而已”，而是保证产品在异常状态下可以恢复到可控状态。

建议写：

> 复位键相当于产品的安全兜底机制。无论当前处于哪个模式，用户都可以通过复位让系统重新进入确定状态，保证演示和日常使用的可恢复性。

### 4.4 校时功能

写作重点：

说明 `SW0=1` 进入设置层，`BTNL / BTNR` 切换字段，`BTNU / BTND` 调整数值，`BTNC` 做确认或上下文操作。仓库 README 也记录了当前交互规则：浏览层用左右键切换模式，设置层用左右键切字段、上下键改数值，提醒激活时普通操作会被锁定，`BTNC` 优先用于确认/消音。

建议写成产品交互语言：

> 我们把校时设计成“浏览层”和“设置层”两层结构。平时用户只浏览时间和模式；需要修改时拨动 `SW0` 进入设置层，再用方向键修改当前字段。这样可以避免误触，也让所有功能共享同一套交互逻辑。

## 5. 统一交互设计：七个模式，一套操作逻辑

### 5.1 模式总览

当前模式顺序建议直接作为一张图或一行流程：

```
CLOCK -> TIME -> ALARM -> HOUR -> COUNT -> SCHED -> COMM -> CLOCK
```

每个模式解释如下：

| 模式    | 产品含义          | 用户看到什么                     |
| ------- | ----------------- | -------------------------------- |
| `CLOCK` | 主时钟 / 日期入口 | 当前时间、日期、星期             |
| `TIME`  | 校时模式          | 时、分、秒编辑                   |
| `ALARM` | 闹钟模式          | 闹钟槽位、开关状态、提醒         |
| `HOUR`  | 显示格式模式      | 12 / 24 小时制切换               |
| `COUNT` | 倒计时模式        | 倒计时设置、运行、停止、到零提醒 |
| `SCHED` | 日程提醒模式      | 固定计划点、槽位开关、最近计划   |
| `COMM`  | PC 通信模式       | ClockLink 连接状态、消息、回复   |

### 5.2 浏览层与设置层

建议写成一个小节：

> 系统使用 `SW0` 区分浏览层和设置层。`SW0=0` 时，用户主要进行模式切换和状态查看；`SW0=1` 时，用户进入当前功能的编辑流程。COMM 模式由于需要使用 `SW0-SW15` 查看最近 16 条消息，因此不再把 `SW0` 解释为设置层开关。

### 5.3 为什么这样设计

这一节可以体现“产品意识”：

> 如果每个功能都单独设计一套按键，用户很快会不知道哪个键在当前模式下代表什么。因此本项目把所有功能抽象成“模式切换、字段切换、数值增减、上下文确认”四类操作，让复杂功能共享一套简单、一致的交互模型。

## 6. 扩展功能介绍：从电子钟到时间管理终端

这一章是加分项重点，建议写得生动一些。

### 6.1 12 / 24 小时制切换

写作重点：

介绍 `HOUR` 模式。它不改变内部 24 小时计时，只改变显示方式。这一点很适合体现“内部数据”和“外部显示”的分层思想。

建议写：

> 系统内部始终采用 24 小时制保存时间，保证闹钟、日程和倒计时比较逻辑统一；用户界面可以切换成 12 小时制显示，从而兼顾逻辑稳定性和用户习惯。

### 6.2 多槽位闹钟

写作重点：

`ALARM` 支持 8 个槽位、LED 状态提示、pending 事件和贪睡。仓库 README 中也明确当前 ALARM 已支持 8 槽位、LED0~LED7 状态提示、pending 事件和闹钟贪睡。

建议写：

> 闹钟功能不只是“到点响一下”，而是按照真实产品思路设计为多槽位管理。用户可以维护多个提醒点，LED 直接展示槽位状态，提醒触发后进入 pending 状态，并支持确认和贪睡处理。

### 6.3 倒计时

写作重点：

`COUNT` 支持 `HH:MM:SS` 编辑、启动、停止和到零提醒。可以强调它与普通时钟不同：一个是向前走时，一个是向后计时，逻辑边界和到零事件处理不同。

建议写：

> 倒计时功能让系统从“看时间”扩展到“管理时间”。用户可以设置一个持续时间，启动后系统自动递减，到零后触发提醒，并由统一提醒模块接管蜂鸣器和 OLED 弹窗。

### 6.4 日程 / 课程提醒

写作重点：

`SCHED` 支持 8 个固定计划点、槽位开关、LED 指示和计划提醒。适合讲“面向课程表/作息提醒”的使用场景。

建议写：

> 日程模式面向课程作息或固定计划提醒。用户可以把一天中需要提醒的时间点保存为计划槽位，系统根据当前时间自动比较，到点后触发提醒。相比单个闹钟，它更像一个轻量级课程表提醒器。

### 6.5 统一提醒系统

写作重点：

`notification_ctrl` 统一仲裁倒计时、闹钟和计划提醒，并作为蜂鸣器唯一驱动源。这个点很重要，体现架构能力。

建议写：

> 多个功能都会产生提醒，如果每个模块都直接驱动蜂鸣器，就会出现冲突。因此项目设计了统一提醒控制器，由它仲裁倒计时、闹钟和日程事件，统一决定蜂鸣器、OLED 弹窗和按键确认行为。这使系统更接近一个完整产品，而不是多个功能的简单拼接。

### 6.6 OLED 状态副屏

写作重点：

OLED 展示日期、温度、最近计划、最近闹钟、倒计时状态和提醒弹窗。COMM 模式下，OLED 显示通信页面、消息正文和预设回复。仓库模块说明中明确 OLED 当前显示日期/星期、温度、最近计划、最近闹钟、倒计时状态、当前模式标识，并在 COMM 模式下切换到通信专用页面。

建议写：

> 数码管负责高可见度的主时间显示，OLED 负责展示更丰富的状态信息。这样的双屏设计让系统既保留电子钟的直观性，又具备智能设备的状态表达能力。

### 6.7 温度显示

写作重点：

板载 ADT7420 温度读取模块已经接入顶层 `TMP_SCL / TMP_SDA`，但 README 里注明板级温度读数尚未实测。文档中可以写“设计已接入”，实测情况按最终验收前结果补充。

建议写法：

> 系统预留并接入了 Nexys A7 板载 ADT7420 温度传感器读取链路，OLED 可作为温度状态显示区域。若最终板级实测完成，可在此处补充温度读数截图；若尚未完成，应如实说明当前为顶层接入和综合通过状态。

## 7. ClockLink Studio：PC 与 FPGA 的通信扩展

这一章是你这个分支最有特色的部分，建议单独成章。

### 7.1 功能定位

可以这样写：

> ClockLink Studio 是本项目的 PC 上位机软件，它让电子钟不再只是板子上的孤立硬件，而是可以通过 USB-UART 与电脑交互的时间管理终端。PC 可以发送消息、同步时间、读取时间、设置闹钟、设置日程、控制倒计时，并查看通信日志。

仓库中的 PC 软件 README 记录了当前已实现协议库、mock FPGA、服务层、CLI demo、Tkinter GUI 演示面板、真实串口 transport 接口和 pytest 单元测试；GUI 包含连接与消息、功能控制、通信日志等区域，并支持中英文界面切换。

### 7.2 通信链路

建议写成：

```
ClockLink Studio
    ↓ USB-UART 115200 8N1
uart_rx / uart_tx
    ↓
protocol_parser / protocol_builder
    ↓
comm_ctrl
    ↓
time_core / alarm_ctrl / schedule_ctrl / countdown_ctrl / message_store
```

最终演示指南中也明确了 USB-UART 参数为 `115200 8N1`，真实串口模式下需要把 `COM5` 替换为 Windows 设备管理器中对应端口；演示流程包括 `HELLO`、`PING/PONG`、`SYNC`、`GET TIME`、发送消息、板上查看消息、预设回复、设置闹钟/日程/倒计时等。

### 7.3 COMM 模式的产品体验

建议写：

> 在 COMM 模式下，数码管左侧显示 `COMM`，右侧显示连接状态，例如 `DISC / WAIT / CONN / MSG! / ERR`。当 PC 发送消息后，板上进入未读消息状态，用户可以通过拨码开关选择最近 16 条消息，通过上下键滚动正文，通过中心键进入回复选择，并用右键发送预设回复。

## 8. 技术实现与课程知识体现

这一章专门回应“老师可能希望看到你学到了什么”。

### 8.1 同步时序设计

写：

> 系统以 `CLK100MHZ` 为唯一主时钟，通过时钟使能产生低速行为，避免随意生成多个内部时钟导致时序难以收敛。这体现了同步时序设计、计数器分频和 FPGA 时钟域管理的基本思想。

### 8.2 动态扫描显示

写：

> Nexys A7 的八位七段数码管需要动态扫描，且位选和段选均为低有效。因此项目将显示内容编码、字符译码和扫描刷新拆成多个模块：`display_ctrl` 负责组织显示字符，`nexys_seg_scan` 负责动态扫描，`seg_7` 负责字符到七段码转换。

### 8.3 按键消抖与单脉冲

写：

> 物理按键不是理想数字信号，按下时会有抖动。因此项目通过按键同步、消抖和单周期脉冲生成，将原始输入转换为可靠的控制事件，再交给 UI 状态机处理。

### 8.4 状态机与模式管理

写：

> 系统有七个工作模式，如果各模块独立读取按键，会造成逻辑混乱。项目使用统一 `ui_ctrl` 管理模式状态、设置层、字段选择、增减脉冲和提醒锁定，使所有功能服从统一交互规则。

### 8.5 外设接口

写：

## 9. 工程优化与稳定性说明

这一章不要写成“我优化了很多”，而是写成“我遇到了什么产品级问题，并如何收敛”。

### 9.1 时序优化

建议写：

> 随着 OLED、COMM、ALARM、SCHED 等功能加入，组合逻辑路径变长，时序压力增大。为此项目采用了若干收敛策略：统一时钟域、使用时钟使能替代乱分内部时钟、将最近闹钟/最近计划选择改为分时扫描并寄存输出、降低 OLED 复杂显示逻辑、将通信构帧拆分为状态过程等。

仓库的模块说明中也提到，ALARM 和 SCHED 最近事件选择已改为分时扫描并寄存输出，OLED 中心模式页已降复杂度以满足 100MHz 综合时序。

### 9.2 WNS / TNS 结果

建议写：

> 当前综合检查结果显示 `WNS=+1.232ns`，`TNS=0.000ns`，失败端点为 `0`。这说明在当前综合检查边界下，设计已经从负时序裕量收敛到正裕量状态，具备继续生成 bitstream 和上板验证的基础。

注意：如果你最后重新跑实现后 WNS 变化，要以最终 Vivado report 为准更新这里。

### 9.3 XDC 与板级约束

写：

> 项目没有全量照抄 Master XDC，而是根据实际使用的顶层端口裁剪项目自己的约束。XDC 的作用是完成管脚绑定和时钟约束，保证 Verilog 逻辑端口能正确映射到 Nexys A7 的真实硬件资源。

### 9.4 验证体系

建议写成三层：

| 验证层次        | 内容                           | 目的                                 |
| --------------- | ------------------------------ | ------------------------------------ |
| 语法 / 展开检查 | `xvlog`、`xelab clock_amd_top` | 确认 HDL 可编译、顶层可展开          |
| 功能仿真        | COMM 相关 testbench            | 确认协议、时间、消息、回复、控制链路 |
| PC 软件测试     | `python -m pytest`             | 确认上位机协议和服务逻辑             |
| 综合时序检查    | Vivado batch synth             | 确认资源和时序初步收敛               |
| 板级实测        | bitstream + Nexys A7           | 确认真实硬件行为                     |

仓库当前 README 记录了 `xvlog`、`xelab`、COMM 回归、PC 端 15 个 pytest 测试和 Vivado 综合检查通过；同时也注明尚未生成 bitstream、尚未完成 Nexys A7 板级 USB-UART/COMM 实测。

## 10. 现场演示流程

这一章可以直接写成老师验收时的操作脚本。

### 10.1 演示准备

写：

1. 打开 Vivado 工程 `clock_amd.xpr`。
2. 确认顶层为 `clock_amd_top.v`。
3. 确认约束文件为 `clock_amd.xdc`。
4. 生成 bitstream 并下载到 Nexys A7。
5. 接好 OLED、蜂鸣器和 USB-UART。
6. 打开 ClockLink Studio。

### 10.2 基础功能演示

建议顺序：

1. 复位系统，观察时间回到初始状态。
2. 进入 `CLOCK` 模式，观察 `HH:MM:SS` 正常走时。
3. 切换到 `TIME` 模式，进入设置层，修改小时、分钟、秒。
4. 返回浏览层，确认修改后的时间继续正常递增。

### 10.3 扩展功能演示

建议顺序：

1. `HOUR`：切换 12 / 24 小时制。
2. `ALARM`：设置一个接近当前时间的闹钟，等待触发，展示 LED / 蜂鸣器 / OLED 弹窗。
3. `COUNT`：设置短倒计时，例如 `00:00:10`，启动、停止、继续，到零后提醒。
4. `SCHED`：设置或展示固定计划点，说明课程作息提醒场景。
5. OLED：展示日期、温度、最近闹钟、最近计划、倒计时状态。

### 10.4 ClockLink Studio 演示

建议顺序：

1. PC 端点击 `HELLO` 建立连接。
2. 点击 `PING`，收到 `PONG`。
3. 点击 `SYNC` 同步 PC 时间到 FPGA。
4. 点击 `GET` 读取 FPGA 时间。
5. 发送消息 `Hello FPGA`，板上 COMM 模式显示 `MSG!`。
6. 在板上用拨码开关选择消息，用按键滚动正文。
7. 板上选择预设回复并发送，PC 日志显示 `REPLY`。
8. PC 端设置闹钟、日程、倒计时，再回到板上查看对应功能是否更新。

最终演示指南中的推荐流程也正是围绕 `PING/PONG`、时间同步、消息发送、预设回复、闹钟/日程/倒计时设置和原功能回归检查展开。
